### 1. The λ_plan controller: naive halve-to-zero → cap-and-hold-at-a-floor (the core fix)

`CanaryController.update()` (`stack/tanitad/train/v4_curriculum.py`) was **down-only with no lower
bound**: a soft canary breach multiplied `_mult` by 0.5 unbounded; three hard breaches set it to **0**.
On v4.1's cut `lr_trunk` any real planner gradient briefly breached the noisy canary, so the ratchet
fired nearly every eval and drove `lam_mult` monotonically to ~0 — **the planner→trunk coupling was
OFF from ~step 2000**. This was **live and MEASURED at the moment v4.1 was stopped (step 16600)**:
`lam_mult = 4.77e-07`, `controller_action = "controller_halved_lambda"` still firing on a canary only
**+0.067** over baseline (0.489 vs 0.421) — i.e. a basically-healthy WM was still starving the planner.

The fix (V4_DESIGN §14.4 O-14 / §5.5 R1 intent; 2026-07-23 synthesis §4/§7): **`_mult` is bounded
below by `mult_floor` (--lam-mult-floor, default 0.25)** at all times. It still only moves DOWN (O-14
preserved: monotone non-increasing) and still never kills (the gate at step 10k is the only kill), but
it **holds at the floor** so once λ_plan has ramped in the planner ALWAYS keeps ≥¼ of its gradient into
the trunk. WM protection now comes from `lr_trunk 1e-4` + the gate, not from sacrificing the planner.

**CPU smoke proof (`--smoke-loop`, MEASURED):** under a forced canary regression the multiplier trace
is **`[0.5, 0.5, 0.5, 0.25, 0.25]`** — halves on the soft breach, drops to the floor 0.25 on three hard
breaches, and **HOLDS at 0.25** (the old code hit `[0.5, 0.5, 0.5, 0.0, 0.0]`). `controller_down_only`,
`controller_held_at_floor`, `controller_never_zero` all true; checkpoint resume restores `_mult`
bit-exact (0.25). Floor of 0 is refused at construction (it *is* the v4.1 bug).

> ⚠️ **In the LIVE run the floor is not yet exercised** — Phase A (steps 0–2000) has scheduled
> `lambda_plan = 0`, so `lam_mult = 1.0` is simply holding (no breach). The floor engages only once the
> Phase-B ramp (2000–8000) pushes λ_plan up and *if* the canary breaches. The cap-and-hold behaviour is
> proven by the code + smoke now; the live confirmation comes at Phase B. This is stated so no one
> over-reads "lam_mult 1.0" as the floor being tested.

### 2. `lr_trunk 1e-4` — between v4's 3e-4 (WM-degrading) and v4.1's 3e-5 (planner-starving)

Deviates from O-14's 3e-4 by Sayed's 2026-07-23 decision. v4's 3e-4 degraded the WM (canary
0.42→1.3 by step 3500); v4.1's 3e-5 protected the WM but (with the halve-to-zero controller) starved
the planner. 1e-4 is 3.3× below the WM-degrading value and 3.3× above the starving value.

### 3. Effective batch 64 (16 micro × 4 accum) — same as v1

v1 (`flagship4b-speedjerk`, registry §1.2) trained at `--batch-size 16 --accum 4` → **effective 64**.
v4.1 ran micro-batch 16 with **accum 1 = effective 16**, 4× smaller (4× less data/step, noisier
gradients — a second contributor to its weak planner). v4.2 wires `--accum 4` into the training loop
(each micro loss scaled by 1/accum, one optimizer step per 4 micro-batches). Micro-batch 16 fits at
**33.2 GB** on the A40 (MEASURED) so grad-checkpoint is not needed. Effective batch 64 is a hard
preflight invariant now.
